# Self-Hosted Toolkit Proposal

## Goal

This repository should evolve into the source-of-truth for the Dvorark toolkit.

It should not be treated as the final starter repository that users will clone directly.
Instead, it should:

- define the CLI application
- define the Electron application
- define the internal generator and graph logic
- define the blueprint files used by `dvorark create`
- prove, through its own structure, that the architectural approach works

The repository created by `dvorark create` will be a new repository, separate from this
one, but structurally aligned with this one.

## Naming: Why Dvorark

The toolkit should be named `Dvorark`, not `Hexavai`.

This name carries the intended identity of the system and should be treated as part of the
architecture, not as superficial branding.

The name is justified by four converging ideas.

### 1. Infinite-State Self-Definition

The strongest conceptual reference is the line associated with Dvorak in _Splinter Cell:
Chaos Theory_: an infinite-state machine that produces a more accurate definition of
itself at every iteration.

That is exactly the intended behavior of this repository and of the generated workspaces.
The system is not meant to be a static pile of folders. Each time the toolkit adds an
entity, use case, module, or wiring rule, the workspace should understand itself better,
validate its own boundaries more accurately, and refine its own shape.

In this sense, Dvorark is not merely the code. Dvorark is the governing algorithm that
drives self-definition.

### 2. Efficiency Against Inertia

The reference to the Dvorak keyboard layout is equally important.

QWERTY survives largely because it is traditional, not because it is the best arrangement.
The toolkit aims to make the same kind of statement about architecture: many software
codebases are arranged the way they are because people are used to it, not because that
structure is ergonomically or cognitively sound.

Dvorark should represent the ergonomic alternative:

- put the domain at the center
- reduce architectural friction
- optimize for clarity and correctness
- help developers write better code, not just faster code

### 3. Isolation And Protection

Another key association is the idea of a protected, isolated machine that is kept immune
from outside interference.

This maps directly to the intended separation between:

- business/product code
- authoring logic such as generators, graph analysis, and orchestration

Dvorark is the protected external mechanism, the armored system that can manipulate and
govern a repository without polluting the repository's business logic with maintenance
scripts and accidental complexity.

### 4. Dvorak + Architecture = Dvorark

The extra `r` is intentional. It reinforces the association with architecture and with the
idea of an ark-like protective vessel.

The name should suggest a system that:

- protects the heart of the codebase
- preserves architectural integrity over time
- transports the domain safely as the repository evolves

In short, Dvorark is intended to be the guardian of the hexagon: a system that knows what
the workspace should be and resists architectural mediocrity.

## Core Principle

This monorepo defines itself.

That means:

- application logic belongs in application-layer packages
- infrastructure logic belongs in infrastructure-layer packages
- composition logic belongs in composition-layer packages
- UI logic belongs in UI-layer packages
- top-level apps compose and expose the toolkit capabilities

The repository should not introduce a custom meta-structure that violates its own
architectural philosophy.

The only major exception is the blueprint area, because blueprint files are effectively
constants and scaffolding assets, not runtime or application logic.

## High-Level Model

There are three distinct things in the system:

1. This repository
2. The blueprint area contained in this repository
3. The new repository generated by `dvorark create`

### 1. This Repository

This repository is the authoring repository of the toolkit.

It contains:

- the `dvorark` CLI app
- the Electron app / Dvorark Studio app
- the package-level logic that powers generation, graphs, and future navigation
- the blueprint files that define the starter skeleton

### 2. The Blueprint Area

The blueprint area contains the files used to create a fresh repository with
`dvorark create`.

It is not part of the runtime toolkit logic.

It is a dedicated area of mostly static assets and Handlebars templates.

Suggested top-level directory names:

- `blueprints/`
- `starter-templates/`

Either is acceptable. At the moment, `blueprints/` is slightly preferable because it is
shorter and reads well in both docs and code.

### 3. The Generated Repository

The generated repository is a fresh workspace created from the blueprint files and the
rules implemented in this repository.

It is not this repository.

It should:

- be structurally similar to this repository
- own its scaffolded product/runtime packages locally
- install and use `dvorark` as a dependency / tool
- be manipulated later through the CLI and the Studio app

## Expected Workflow

The intended workflow is:

1. Work on toolkit code in this repository
2. Maintain blueprint files in a dedicated blueprint area in this repository
3. Run `dvorark create` to generate a new, separate repository
4. Use `dvorark` CLI and Studio inside that generated repository
5. Evolve this repository whenever the toolkit, blueprint rules, or starter structure change

In short:

- this repo is the authoring and proving ground
- generated repos are the actual starter instances

## Boundary Between Toolkit Logic and Product Scaffolding

This distinction is essential.

### Toolkit Logic

Toolkit logic includes:

- workspace scanning
- graph building
- graph rendering orchestration
- code generation orchestration
- AST transforms
- wiring logic
- demo orchestration logic
- CLI command handling
- Electron-side orchestration and UI interactions

This logic belongs in real packages and apps under the monorepo architecture.

### Product Scaffolding

Product scaffolding includes:

- root files for a newly created repository
- package skeleton files
- starter package manifests
- starter config files
- starter app/package source placeholders
- `.hbs` templates and static blueprint files

This content should live outside `packages/`, in a dedicated top-level blueprint area.

## Blueprint Area

The blueprint area should be a dedicated top-level directory, not a package.

Recommended shape:

```text
blueprints/
  starter/
    root/
    configs/
    packages/
    apps/
```

Alternative acceptable name:

```text
starter-templates/
  workspace/
```

The important rule is not the exact name; it is that the area is:

- outside `packages/`
- clearly recognized as scaffolding assets
- mostly static
- not confused with toolkit runtime logic

## Template Technology

Handlebars remains a good fit for blueprint rendering.

The recommendation is:

- keep scaffolding artifacts as `.hbs` where templating is needed
- keep static files as plain files where no interpolation is needed
- avoid pushing template logic too far into the templates themselves
- keep real logic in TypeScript packages, not inside Handlebars expressions

This preserves a useful distinction:

- `.hbs` = blueprint assets
- TypeScript = actual behavior and orchestration

## Agent Access And MCP

The toolkit should also expose an MCP surface so that coding agents can invoke the same
high-level capabilities that are available through the CLI.

This is important because the CLI should become the main operational API of the toolkit,
and agents should not have to reimplement repository-specific knowledge in prompts or
ad-hoc scripts.

The intended direction is:

- the CLI defines the user-facing operational commands
- an MCP server exposes those operations in a structured way for agents
- the Electron app can reuse the same underlying internal logic where appropriate

The MCP layer should not become a separate source of truth. It should be an adapter on top
of the same internal packages used by the CLI and Studio.

Examples of capabilities that should eventually be invokable through MCP:

- create a new repository from the blueprint area
- run generation workflows
- build package/composition/module graphs
- inspect workspace structure
- trigger demo generation or removal

This gives the toolkit three coherent access modes:

- human CLI
- human UI
- agent MCP

All three should converge on the same internal behavior.

## Skills And Paradigm Teaching

Besides executable tooling, the repository should also include Markdown-based skills or
guides that teach the Hexagonal+VAI paradigm and the intended use of the toolkit.

This material is useful for both humans and agents.

The role of this material is to explain:

- the architectural philosophy
- the layer responsibilities
- naming and workspace conventions
- how generation and wiring are expected to work
- how graphs should be interpreted
- what the standard path is and what happens when a repository opts out

Recommended characteristics:

- Markdown-first
- concise and explicit
- versioned with the toolkit
- written as operational guidance, not marketing copy

Possible forms:

- `docs/skills/*.md`
- a dedicated `skills/` area at the repository root
- blueprint-facing guidance plus toolkit-internal guidance

The important rule is that the paradigm should not live only in code and conversation.
It should also be available as durable written guidance that can be consumed by humans,
agents, and future UI help surfaces.

## Proposed Monorepo Roles

### Top-Level Apps

Two first-class apps should be introduced:

- `apps/dvorark-cli`
- `apps/dvorark-studio`

#### `apps/dvorark-cli`

This app replaces:

- Plop as the primary generation entrypoint
- custom graph scripts as the public entrypoint

Its role is to expose commands such as:

- create
- generate
- graph package
- graph composition
- graph module
- demo generate
- demo remove

#### `apps/dvorark-studio`

This Electron app exposes the same core capabilities through a user-friendly interface.

Its role is to provide:

- graph visualization
- future interactive graph navigation
- generation workflows
- preview and guidance
- repository navigation support
- future editor integration support

It should not contain the core business logic of generation or graph building. It should
consume internal packages that expose that behavior.

## Package Placement Philosophy

The repository should continue to demonstrate its own architectural philosophy.

That means package logic should live under the standard layers, according to meaning.

No dedicated `tooling` layer should be introduced.

If the CLI needs application-level orchestration, that orchestration should live in
application-layer packages.

If graph rendering or filesystem access are infrastructure concerns, they should live in
infrastructure-layer packages.

If the CLI or Studio need composition roots, those should live in composition-layer
packages.

If the Studio needs UI packages, those should live in UI-layer packages.

This is intentional self-hosting.

## Structural Consequence

Because of the self-hosting rule, the monorepo should be treated as:

- a real consumer of its own architectural conventions
- not a special toolkit-only codebase with custom exceptions

The generated repository should then mirror the same architectural structure, but without
the authoring-only toolkit code and without the internal blueprint source area.

## Relationship Between This Repo and Generated Repos

The generated repo should be structurally aligned with this repo, but not identical in
content.

### This Repo Contains

- toolkit source code
- authoring apps
- graph logic
- generator logic
- demo logic
- blueprint assets

### Generated Repo Contains

- starter structure derived from the blueprints
- scaffolded product/runtime packages
- workspace configs
- no authoring-only blueprint source area
- no need to include the source code of this repository's internal toolkit implementation

The generated repo should use the published or installed toolkit, not embed all toolkit
source directly.

## Runtime / Scaffolded Packages In Generated Repos

Generated repositories should scaffold and own their standard local packages, including:

- `@domain/core`
- `@application/core`
- `@infrastructure/lib-http`
- `@infrastructure/lib-dataloader`

These packages are product/runtime-level starter material.

They are not meant to remain hidden behind opaque toolkit dependencies.

If a generated repo later modifies, removes, or breaks them, that repository is
effectively opting out of the standard path, and some generator features may stop working.
That is acceptable.

## Role Of Plop Going Forward

Plop can remain a temporary implementation detail if needed during migration.

However, it should no longer be treated as the target architecture.

Target direction:

- a dedicated CLI app as the public interface
- dedicated internal packages for generation logic
- optional short-term compatibility layers while migrating

In other words:

- Handlebars may remain
- Plop should become replaceable

## Role Of Existing Graph Scripts

Existing graph scripts should be considered transitional entrypoints.

The target state is:

- graph behavior exposed through internal packages
- public graph operations exposed through `apps/dvorark-cli`
- the Studio app reusing the same underlying graph logic

When migration is complete:

- custom root scripts for graphs can be removed
- public entrypoints are the CLI and the Studio app

## Electron / Studio Scope

The Studio app should be treated as a command center, not as a replacement for the IDE.

Its mission is:

- architecture navigation
- graph visualization
- generation workflows
- guided understanding of the workspace
- future ability to jump to files in the user's editor

It should complement the IDE, not compete with it.

## Suggested Phased Migration

The repository should evolve in phases rather than through a big-bang rewrite.

### Phase 1

- create `docs/` and document the target architecture
- introduce blueprint area
- introduce CLI app skeleton
- introduce Studio app skeleton

### Phase 2

- extract graph logic from root scripts into internal packages
- expose graph operations through the CLI
- keep legacy scripts temporarily as thin wrappers if necessary

### Phase 3

- extract generation logic away from Plop-centric entrypoints
- keep Handlebars templates in the blueprint area
- expose generation through the CLI

### Phase 4

- wire the Studio app to the same graph/generator logic
- add interactive graphs and guided flows

### Phase 5

- remove Plop as a first-class dependency
- remove legacy root graph scripts
- keep only the new public surfaces

## Non-Goals For Now

The following should not be prioritized in the first iteration:

- making layer names configurable
- making scopes configurable
- making naming conventions configurable
- introducing a custom meta-layer only for tooling
- turning the Studio app into a full IDE

The system should remain opinionated.

## Summary

The proposal is:

- keep this repository as the self-hosted authoring repository
- add two real apps: CLI and Studio
- keep architecture self-consistent by placing logic in the standard layers
- keep scaffolding assets outside `packages/` in `blueprints/` or `starter-templates/`
- continue using Handlebars for scaffolding where useful
- migrate away from Plop and root graph scripts gradually
- generate a separate repository with `dvorark create`
- use `dvorark` CLI and Studio inside that generated repository

This preserves the repository's philosophy while giving it a clean path toward a more
maintainable and extensible toolkit architecture.
